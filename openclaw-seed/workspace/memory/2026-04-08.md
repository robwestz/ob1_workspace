## Knowledge Base Architecture
Three layers, each serves a different purpose:
1. **Domain knowledge** — what we know about specific fields (SEO, link building, agent systems)
2. **Methods** — how we do things (wave protocol, QA gates, deployment patterns)
3. **Components** — reusable building blocks (runtime modules, Edge Functions, dashboard components)

Plus: _templates/ for standardized document formats, _system/ for KB management config, raw/ for unprocessed research.

## OB1 Control — 10 Phases
1. SysAdmin identity (persistent persona with memory and goals)
2. Multi-model gateway (Claude, Codex, Gemini dispatch)
3. CLI (`ob1 status`, `ob1 night start`, `ob1 deploy`)
4. Wave-runner integration (automated overnight execution)
5. Overnight sessions (true 7.5h runs with budget enforcement)
6. Deploy pipeline (Windows push -> Mac pull -> build -> restart)
7. Dashboard (localhost:4000, service health, morning reports)
8. Initiative system (agents propose improvements, not just execute)
9. Self-improvement loop (SysAdmin improves its own tooling)
10. Integration (everything connected: CLI -> runtime -> dashboard -> Supabase -> Mac)

## OpenClaw Decision
Considered building our own multi-channel gateway. Decided against it — OpenClaw already handles WhatsApp, Telegram, Discord, Slack with auth, session management, and a mature codebase. We integrate with it instead of competing. OpenClaw runs on the Mac, our runtime talks to it via gateway API.
